## Задача 1. Базовое применение Quick-Select

Дан массив `A = [5, 2, 8, 3, 9, 1, 7, 4, 6]`. Найдите 4-й наименьший элемент, выполнив одну итерацию Quick-Select с пивотом `A[4] = 9`. Покажите состояние массива после разбиения Lomuto и объясните, куда рекурсировать.

### Решение

Выбираем пивот `p = 9` (элемент с индексом 4), перемещаем в конец: `A = [5, 2, 8, 3, 1, 7, 4, 6, 9]`.

Применяем разбиение Lomuto (все элементы $\leq 9$ собираем влево, `i = -1` изначально):

| j | A[j] | $\leq 9$? | Действие | i |
|---|------|-----------|----------|---|
| 0 | 5 | да | swap(A[0], A[0]), i=0 | 0 |
| 1 | 2 | да | swap(A[1], A[1]), i=1 | 1 |
| 2 | 8 | да | swap(A[2], A[2]), i=2 | 2 |
| 3 | 3 | да | swap(A[3], A[3]), i=3 | 3 |
| 4 | 1 | да | swap(A[4], A[4]), i=4 | 4 |
| 5 | 7 | да | swap(A[5], A[5]), i=5 | 5 |
| 6 | 4 | да | swap(A[6], A[6]), i=6 | 6 |
| 7 | 6 | да | swap(A[7], A[7]), i=7 | 7 |

После прохода: swap(`A[8]`, `A[8]`) — пивот 9 уже в конце, встаёт на позицию $p = 8$.

Массив: `[5, 2, 8, 3, 1, 7, 4, 6, 9]`.

Ищем $k = 3$ (0-индексация, 4-й наименьший). Пивот на позиции $p = 8 > k = 3$ — рекурсируем в левую часть `[5, 2, 8, 3, 1, 7, 4, 6]` (индексы 0–7), ищем 4-й наименьший там.

(Ответ: 4, отсортированный массив `[1, 2, 3, 4, 5, 6, 7, 8, 9]`.)

### Ответ

После одной итерации пивот 9 встаёт на позицию 8 (последнюю). Рекурсируем в `[5, 2, 8, 3, 1, 7, 4, 6]`, ищем 4-й наименьший. Итоговый ответ: **4**.

## Задача 2. Трассировка разбиения Lomuto

Массив `[4, 7, 2, 9, 1, 5, 3, 8]`, пивот = `A[7] = 8` (уже в конце). Выполните разбиение Lomuto вручную. Укажите позицию пивота после разбиения и содержимое массива.

### Решение

Пивот $x = 8$, массив `[4, 7, 2, 9, 1, 5, 3, 8]`, $i = -1$.

- $j=0$: $A[0]=4 \leq 8$ — $i=0$, swap($A[0], A[0]$) — без изменений
- $j=1$: $A[1]=7 \leq 8$ — $i=1$, swap($A[1], A[1]$) — без изменений
- $j=2$: $A[2]=2 \leq 8$ — $i=2$, swap($A[2], A[2]$) — без изменений
- $j=3$: $A[3]=9 > 8$ — пропуск
- $j=4$: $A[4]=1 \leq 8$ — $i=3$, swap($A[3], A[4]$): массив `[4, 7, 2, 1, 9, 5, 3, 8]`
- $j=5$: $A[5]=5 \leq 8$ — $i=4$, swap($A[4], A[5]$): массив `[4, 7, 2, 1, 5, 9, 3, 8]`
- $j=6$: $A[6]=3 \leq 8$ — $i=5$, swap($A[5], A[6]$): массив `[4, 7, 2, 1, 5, 3, 9, 8]`

Финальный swap($A[6], A[7]$): массив `[4, 7, 2, 1, 5, 3, 8, 9]`. Пивот на позиции $p = 6$.

Проверка: $A[0..5] = [4, 7, 2, 1, 5, 3]$ — все $\leq 8$. $A[7] = 9 > 8$. Верно.

### Ответ

Результат: `[4, 7, 2, 1, 5, 3, 8, 9]`, пивот 8 на позиции **6**.

## Задача 3. Количество сравнений в Quick-Select

Оцените ожидаемое число сравнений ключей в рандомизированном Quick-Select при поиске минимума в массиве из $n$ элементов. Почему оно равно $2(n-1)$, а не $n-1$?

### Решение

При поиске минимума ($k=0$) каждый раз нужно полностью пройти текущий отрезок. Рассмотрим вклад каждой пары элементов $(i, j)$ в ожидаемое число сравнений.

В Quick-Select элементы $i$ и $j$ сравниваются тогда и только тогда, когда один из них становится пивотом до того, как они оказались в разных частях. Вероятность этого события для двух конкретных элементов с рангами $r_i < r_j$:

$$
\Pr[\text{i и j сравниваются}] = \frac{2}{r_j - r_i + 1}
$$

Суммируя по всем парам и учитывая, что ищем минимум (ранг 0), вклад любой пары $(0, j)$ равен $2/j$:

$$
E[\text{сравнений}] = \sum_{j=1}^{n-1} \frac{2}{j+1} \cdot \text{...} \approx 2n
$$

Более простой аргумент: при поиске минимума в массиве из $n$ элементов нужно минимум $n-1$ сравнений (нижняя оценка). В Quick-Select каждый проход по подмассиву длины $m$ делает $m-1$ сравнений. Ожидаемая суммарная длина всех подмассивов — около $2n$ из-за возможного плохого разбиения в ранних итерациях. Точный анализ даёт $E \approx 2(n-1)$.

Причина отличия от $n-1$: Quick-Select может выбирать плохие пивоты, и тогда проходит почти весь массив несколько раз, прежде чем найдёт нужный элемент.

### Ответ

Ожидаемое число сравнений $\approx 2(n-1)$. Это линейно, но с коэффициентом 2, а не 1, из-за возможных плохих разбиений на ранних шагах алгоритма.

## Задача 4. Группы по 3 в медиане медиан

Покажите, что если в алгоритме медианы медиан использовать группы по 3 вместо 5, то алгоритм перестаёт работать за $O(n)$. Выпишите рекуррентность и решите её.

### Решение

При группах по 3:
- Число групп: $\lceil n/3 \rceil$.
- Рекурсия для поиска медианы медиан: $T(n/3)$.
- Гарантия разбиения: медиана $M$ гарантирует, что хотя бы $\lceil n/3 \rceil / 2 \approx n/6$ медиан $\geq M$, и в каждой такой группе из 3 ещё 1 элемент $\geq M$. Итого элементов $\geq M$: $\approx 2 \cdot n/6 = n/3$.
- Значит, большая часть содержит $\leq n - n/3 = 2n/3$ элементов.

Рекуррентность:

$$
T(n) \leq T\!\left(\frac{n}{3}\right) + T\!\left(\frac{2n}{3}\right) + O(n)
$$

Проверяем подстановкой $T(n) \leq cn$:

$$
T(n) \leq \frac{cn}{3} + \frac{2cn}{3} + an = cn + an
$$

Нам нужно $cn + an \leq cn$ — это невозможно при $a > 0$! Подстановка не работает.

Используем Master theorem. Это не стандартная форма, но воспользуемся методом Акра–Баацци или оценим сверху. Заменим $T(2n/3) \leq T(n \cdot 2/3)$:

$$
T(n) = T(n/3) + T(2n/3) + O(n)
$$

Пусть $f(n) = n$ и оба поддерева. На $k$-м уровне рекурсии суммарный размер работы: вычислим «вес» дерева. На уровне 0: $n$. На уровне 1: $n/3 + 2n/3 = n$. Каждый уровень даёт $O(n)$ работы. Глубина дерева: максимальная ветвь идёт по $2n/3$, глубина $= \log_{3/2} n = O(\log n)$.

$$
T(n) = O(n \log n)
$$

### Ответ

Рекуррентность $T(n) = T(n/3) + T(2n/3) + O(n)$ даёт $T(n) = O(n \log n)$ — алгоритм перестаёт быть линейным.

## Задача 5. Доказательство корректности 30%/70%

Докажите строго, что медиана медиан $M$ при группах по 5 гарантирует, что в каждой части разбиения не более $7n/10$ элементов.

### Решение

Обозначим $m = \lceil n/5 \rceil$ — число групп (считаем все полными для простоты; неполная последняя группа только уменьшает оценку).

**Утверждение:** не менее $3n/10$ элементов массива $\geq M$.

**Доказательство:**

1. $M$ — медиана $m$ медиан, значит хотя бы $\lceil m/2 \rceil \geq m/2$ медиан $\geq M$.

2. Для каждой такой медианы $m_i \geq M$: в группе из 5 есть как минимум 3 элемента (медиана и два больших) $\geq m_i \geq M$.

3. Итого элементов $\geq M$:

$$
3 \cdot \left\lceil \frac{m}{2} \right\rceil \geq 3 \cdot \frac{m}{2} = \frac{3m}{2} \geq \frac{3}{2} \cdot \frac{n}{5} = \frac{3n}{10}
$$

**Следствие:** элементов $< M$ не более $n - 3n/10 = 7n/10$. То есть при разбиении по $M$ правая часть (элементы $> M$) содержит не более $7n/10$ элементов, и левая часть тоже не более $7n/10$.

Поэтому рекурсия идёт в подмассив размером $\leq 7n/10$.

### Ответ

Формально: число элементов $\geq M$ не менее $3\lceil m/2 \rceil \geq 3n/10$. Значит каждая из двух частей содержит не более $n - 3n/10 = 7n/10$ элементов. Гарантия разбиения $30\%/70\%$ доказана.

## Задача 6. Master theorem для порядковых статистик

Решите рекуррентность $T(n) = T(n/5) + T(7n/10) + cn$, показав, что $T(n) = O(n)$ методом подстановки. Найдите минимальную константу $C$, при которой $T(n) \leq Cn$.

### Решение

Предположим $T(n) \leq Cn$ для всех $< n$. Подставляем:

$$
T(n) \leq C \cdot \frac{n}{5} + C \cdot \frac{7n}{10} + cn = Cn\!\left(\frac{1}{5} + \frac{7}{10}\right) + cn = Cn \cdot \frac{2+7}{10} + cn = \frac{9Cn}{10} + cn
$$

Нужно: $\frac{9Cn}{10} + cn \leq Cn$.

$$
cn \leq Cn - \frac{9Cn}{10} = \frac{Cn}{10}
$$

$$
c \leq \frac{C}{10} \implies C \geq 10c
$$

При $C = 10c$ неравенство обращается в равенство — база индукции (для малых $n$, где $T(n) = O(1) \leq Cn$) также выполняется, если взять достаточно большое $C$.

Таким образом, **минимальная константа** $C = 10c$, где $c$ — константа в $O(n)$-члене рекуррентности.

### Ответ

$T(n) \leq 10c \cdot n = O(n)$. Минимальная константа в оценке $Cn$ равна $C = 10c$.

## Задача 7. Нижняя оценка: $\Omega(n)$ для выбора

Докажите, что любой алгоритм нахождения $k$-й порядковой статистики в произвольном несортированном массиве требует $\Omega(n)$ сравнений в худшем случае.

### Решение

Рассмотрим алгоритм, который нашёл $k$-й элемент $x$, просмотрев менее $n-1$ элементов, то есть не просматривая хотя бы один элемент $y$.

**Аргумент противника.** Противник выбирает входные данные адаптивно. Пока алгоритм не сравнил $x$ и $y$, противник может изменить значение $y$:

- Если $y$ не просматривался, противник делает его меньше $x$ или больше $x$ — в зависимости от того, что изменит ранг $x$.

Конкретно: пусть алгоритм выдал ответ $x$ как $k$-й наименьший, не просмотрев элемент $y$. Противник назначает $y$ значение, которое оказывается $< x$ — тогда ранг $x$ увеличивается, и $x$ больше не является $k$-м. Ответ алгоритма был неверным.

Формально: чтобы определить $k$-й элемент, алгоритм должен убедиться, что $k-1$ элементов меньше него и $n-k$ элементов больше. Это требует $n-1$ сравнений в худшем случае.

**Информационный аргумент:** входных конфигураций $n!$, и каждое сравнение делает не более двух исходов — отсюда нижняя оценка $\lceil \log_2 n! \rceil = \Omega(n \log n)$ для сортировки. Для выбора $k$-й статистики — аналогично, но нужно различить лишь $n$ исходов (какой из $n$ элементов является ответом), отсюда $\Omega(\log n)$ информационная оценка. Однако через аргумент противника получаем строгую нижнюю оценку $\Omega(n)$.

### Ответ

Любой алгоритм должен рассмотреть каждый элемент хотя бы один раз, иначе противник изменит непросмотренный элемент и сделает ответ неверным. Поэтому нижняя оценка — $\Omega(n)$ сравнений.

## Задача 8. Quick-Select на связном списке

Можно ли применить Quick-Select к односвязному списку так же эффективно, как к массиву? Если нет, предложите оптимальный алгоритм поиска $k$-й порядковой статистики в односвязном списке.

### Решение

**Почему Quick-Select не применяется напрямую:**

Quick-Select требует операции «обмен элементов» и произвольного доступа (swap по индексу). В односвязном списке:
1. Нет произвольного доступа — доступ к $i$-му элементу стоит $O(i)$.
2. Разбиение Lomuto требует переставлять элементы, что в списке неэффективно (нужно перелинковать узлы).

Таким образом, наивная реализация Quick-Select на списке даёт $O(n^2)$ даже при хорошем пивоте.

**Оптимальный алгоритм для списка:**

1. **Скопировать в массив:** перебрать список ($O(n)$), скопировать значения в `std::vector`. Применить Quick-Select к массиву за $O(n)$. Итого: $O(n)$.

2. **Частичное копирование (экономия памяти):** если памяти мало, можно итерировать список $O(k)$ раз, поддерживая «кучу» из $k$ элементов — $O(n \log k)$.

3. **Partition на месте (перелинковка):** теоретически разбиение на списке делается за $O(n)$ перелинковкой узлов, не переставляя данные. Тогда Quick-Select на списке работает за $O(n)$ в среднем, но реализация значительно сложнее.

**Вывод:** стандартное решение — скопировать в массив и применить Quick-Select или `std::nth_element`.

### Ответ

Quick-Select неэффективен на связном списке из-за отсутствия произвольного доступа. Оптимально — скопировать в массив за $O(n)$, затем применить Quick-Select за $O(n)$. Итого: $O(n)$ время, $O(n)$ память.

## Задача 9. Взвешенная медиана

**Определение.** Взвешенная медиана массива $(x_1, w_1), ..., (x_n, w_n)$ — это элемент $x_k$ такой, что:

$$
\sum_{x_i < x_k} w_i < \frac{1}{2} \quad \text{и} \quad \sum_{x_i > x_k} w_i < \frac{1}{2}
$$

где $\sum w_i = 1$. Предложите алгоритм нахождения взвешенной медианы за $O(n)$.

### Решение

**Шаг 1.** Найдём обычную (невзвешенную) медиану $m$ массива значений $x_i$ за $O(n)$ с помощью Quick-Select.

**Шаг 2.** Вычислим:
- $W_L = \sum_{x_i < m} w_i$ — суммарный вес элементов левее медианы
- $W_R = \sum_{x_i > m} w_i$ — суммарный вес элементов правее медианы
- $W_m = w_m$ — вес медианы

**Шаг 3.**
- Если $W_L < 1/2$ и $W_R < 1/2$: обычная медиана $m$ является взвешенной медианой. Готово.
- Если $W_L \geq 1/2$: взвешенная медиана находится левее $m$. Рекурсируем в левую часть, но добавляем $W_m + W_R$ к весам левых элементов (нормируем).
- Если $W_R \geq 1/2$: аналогично рекурсируем вправо.

**Анализ:** на каждом шаге размер задачи уменьшается минимум вдвое (так как берём часть, содержащую медиану по числу элементов). Работа на шаге $i$: $O(n/2^i)$. Итого: $O(n) \cdot \sum_{i=0}^{\infty} (1/2)^i = O(2n) = O(n)$.

**Альтернатива:** сразу применить взвешенный Quick-Select, где «хорошесть» пивота оценивается по суммарному весу, а не числу элементов.

### Ответ

Алгоритм: найти обычную медиану Quick-Select'ом за $O(n)$; сравнить суммы весов слева и справа; рекурсировать в ту часть, где суммарный вес $\geq 1/2$. Итоговая сложность: $O(n)$.

## Задача 10. Выбор $k$-го элемента из $k$ отсортированных массивов

Даны $k$ отсортированных массивов, каждый длины $n$. Суммарно $kn$ элементов. Предложите алгоритм нахождения медианы всех $kn$ элементов (то есть $\lfloor kn/2 \rfloor$-го наименьшего) без слияния массивов.

### Решение

Применим бинарный поиск по ответу + подсчёт ранга.

**Ключевое наблюдение:** для любого значения $v$ мы можем за $O(k \log n)$ подсчитать, сколько из $kn$ элементов меньше $v$ (бинарный поиск в каждом из $k$ массивов).

**Алгоритм:**

1. Найдём диапазон значений: $[\min, \max]$ всех элементов. Если значения целые, применяем бинарный поиск по значению.

2. Для каждого $v$: $\text{rank}(v) = \sum_{i=1}^{k} |\{x \in A_i : x \leq v\}|$, что вычисляется за $O(k \log n)$.

3. Бинарным поиском по $v$ находим наименьшее $v$ с $\text{rank}(v) \geq kn/2$: $O(\log(\max - \min))$ итераций.

**Итог:** $O(k \log n \cdot \log(\max - \min))$.

**Альтернатива без предположения о диапазоне (рандомизированная):**

1. Выбрать случайный элемент из произвольного массива как пивот $p$.
2. Подсчитать его ранг среди всех $kn$ элементов: $O(k \log n)$.
3. Если ранг $= kn/2$ — готово. Если меньше — медиана правее (отсечь часть каждого массива). Если больше — левее.
4. Каждый шаг с вероятностью $\geq 1/2$ уменьшает число оставшихся элементов минимум на $1/4$.

Ожидаемое число шагов $O(\log(kn))$, каждый стоит $O(k \log n)$.

**Ожидаемое время:** $O(k \log n \cdot \log(kn))$.

### Ответ

Оптимальный алгоритм: бинарный поиск по значению + подсчёт ранга. Сложность: $O(k \log n \cdot \log(\max - \min))$ для целых чисел. Для общего случая: рандомизированный QuickSelect по объединённому пространству за $O(k \log n \cdot \log(kn))$ в среднем.
