## Задача 1. Вычисление хеша и ручная вставка

Вставьте ключи $\{5, 28, 19, 15, 20, 33, 12, 17, 10\}$ в хеш-таблицу с $m = 9$ слотами, используя хеш-функцию $h(k) = k \bmod 9$ и метод цепочек. Нарисуйте итоговую таблицу. Вычислите коэффициент загрузки.

### Решение

Применяем $h(k) = k \bmod 9$ к каждому ключу:

| Ключ | $k \bmod 9$ |
|------|------------|
| 5    | 5          |
| 28   | 1          |
| 19   | 1          |
| 15   | 6          |
| 20   | 2          |
| 33   | 6          |
| 12   | 3          |
| 17   | 8          |
| 10   | 1          |

Итоговые цепочки (элементы вставляются в начало списка):

- Слот 0: пусто
- Слот 1: [10 → 19 → 28]
- Слот 2: [20]
- Слот 3: [12]
- Слот 4: пусто
- Слот 5: [5]
- Слот 6: [33 → 15]
- Слот 7: пусто
- Слот 8: [17]

Коэффициент загрузки: $\alpha = n/m = 9/9 = 1{,}0$.

Ожидаемое время поиска: безуспешного — $\Theta(1 + 1) = \Theta(2)$, успешного — $\Theta(1 + 0{,}5) = \Theta(1{,}5)$.

### Ответ

Цепочки: слот 1: [10→19→28], слот 2: [20], слот 3: [12], слот 5: [5], слот 6: [33→15], слот 8: [17]. Коэффициент загрузки $\alpha = 1{,}0$.

## Задача 2. Выбор размера хеш-таблицы

Объясните, почему размер хеш-таблицы $m = 1024$ — плохой выбор при использовании хеш-функции $h(k) = k \bmod m$ для целочисленных ключей. Предложите лучшую альтернативу. Что изменится, если ключи — адреса памяти, выровненные по 8 байт?

### Решение

$m = 1024 = 2^{10}$. Операция $k \bmod 2^{10}$ выбирает только 10 младших битов числа $k$, игнорируя старшие биты. Если старшие биты несут значимую информацию (например, отличают разные сущности), хеш-функция их полностью игнорирует. Распределение ключей по слотам определяется только последними 10 битами.

Для адресов памяти, выровненных по 8 байт: все адреса кратны 8, значит 3 младших бита всегда равны нулю. При $m = 1024$ хеш будет принимать только значения, кратные 8: $\{0, 8, 16, ..., 1016\}$ — задействованы лишь 128 из 1024 слотов, остальные 896 всегда пусты. Коэффициент загрузки реальной части таблицы вырастает в 8 раз.

Лучшая альтернатива: выбрать простое число $m$ вблизи нужного размера. Например, вместо 1024 взять $m = 1021$ (простое). Тогда хеш использует все биты числа и равномерно распределяет адреса, кратные 8 (так как $\gcd(8, 1021) = 1$).

### Ответ

$m = 2^{10}$ использует только 10 младших битов; для адресов кратных 8 — лишь 128 из 1024 слотов. Решение: $m = 1021$ (простое число).

## Задача 3. Парадокс дней рождения

В хеш-таблице $m = 10000$ слотов. Сколько ключей нужно вставить, чтобы вероятность хотя бы одной коллизии превысила 50%? Запишите формулу и вычислите приближённый ответ.

### Решение

Вероятность того, что первые $n$ ключей попадут в разные слоты:

$$
P(\text{нет коллизии}) = 1 \cdot \frac{m-1}{m} \cdot \frac{m-2}{m} \cdots \frac{m-n+1}{m} = \prod_{i=0}^{n-1} \frac{m-i}{m}
$$

Используем приближение $\ln(1-x) \approx -x$ при малых $x$:

$$
\ln P(\text{нет коллизии}) = \sum_{i=0}^{n-1} \ln\!\left(1 - \frac{i}{m}\right) \approx -\sum_{i=0}^{n-1} \frac{i}{m} = -\frac{n(n-1)}{2m}
$$

Условие $P(\text{нет коллизии}) = 0{,}5$:

$$
e^{-n(n-1)/(2m)} = 0{,}5 \implies \frac{n(n-1)}{2m} = \ln 2 \implies n \approx \sqrt{2m \ln 2}
$$

При $m = 10000$:

$$
n \approx \sqrt{2 \cdot 10000 \cdot 0{,}693} \approx \sqrt{13863} \approx 118
$$

Проверка: $\sqrt{m} = 100$, наш ответ $\approx 118 \approx 1{,}18\sqrt{m}$ — согласуется с правилом $n \approx \sqrt{m}$.

### Ответ

$n \approx \sqrt{2m \ln 2} \approx 118$ ключей.

## Задача 4. Доказательство ожидаемой длины цепочки

Докажите, что при гипотезе простого равномерного хеширования ожидаемая длина цепочки в любом фиксированном слоте $j$ равна $\alpha = n/m$.

### Решение

Зафиксируем слот $j$. Для каждого ключа $k_i$ ($i = 1, \ldots, n$) введём индикаторную случайную величину:

$$
X_{ij} = \begin{cases} 1 & \text{если } h(k_i) = j \\ 0 & \text{иначе} \end{cases}
$$

По гипотезе SUHA каждый ключ равновероятно попадает в любой из $m$ слотов:

$$
P(X_{ij} = 1) = \frac{1}{m}
$$

Следовательно:

$$
\mathbb{E}[X_{ij}] = \frac{1}{m}
$$

Длина цепочки в слоте $j$ — это число ключей, попавших в этот слот:

$$
L_j = \sum_{i=1}^{n} X_{ij}
$$

По линейности математического ожидания:

$$
\mathbb{E}[L_j] = \sum_{i=1}^{n} \mathbb{E}[X_{ij}] = \sum_{i=1}^{n} \frac{1}{m} = \frac{n}{m} = \alpha
$$

Заметим, что мы нигде не использовали независимость $X_{ij}$ между собой — линейность ожидания работает для любых случайных величин. Независимость (из SUHA) нужна для вычисления дисперсии и более точных оценок.

### Ответ

$\mathbb{E}[L_j] = \alpha$. Доказательство через индикаторные СВ и линейность ожидания.

## Задача 5. Анализ времени успешного поиска

При гипотезе SUHA покажите, что ожидаемое время успешного поиска в хеш-таблице с цепочками равно $\Theta(1 + \alpha/2)$.

### Решение

Предположим, что $n$ ключей $k_1, k_2, \ldots, k_n$ вставлены в таблицу в этом порядке, и мы ищем равномерно случайный ключ $k_i$.

При вставке $k_i$ в голову списка, после вставки всех $n$ ключей, $k_i$ находится за $n - i$ элементами (теми, что были вставлены после). При поиске $k_i$ нужно пройти по цепочке до него — это $(\text{число элементов, вставленных в тот же слот после } k_i) + 1$ шагов.

Для ключей $k_j$ с $j > i$: вероятность $P(h(k_j) = h(k_i)) = 1/m$ по SUHA. Ожидаемое число таких ключей: $(n - i)/m$.

Итого, ожидаемое число шагов при поиске $k_i$:

$$
\mathbb{E}[\text{шаги}(k_i)] = 1 + \frac{n - i}{m}
$$

Усредняем по всем равновероятным $i = 1, \ldots, n$:

$$
\mathbb{E}[\text{успешный поиск}] = \frac{1}{n} \sum_{i=1}^{n} \left(1 + \frac{n-i}{m}\right) = 1 + \frac{1}{nm} \sum_{i=1}^{n}(n-i)
$$

$$
= 1 + \frac{1}{nm} \cdot \frac{n(n-1)}{2} = 1 + \frac{n-1}{2m} \approx 1 + \frac{\alpha}{2} = \Theta(1 + \alpha)
$$

(при $\alpha = O(1)$ оба выражения $\Theta(1)$; точная форма $1 + (n-1)/(2m)$, что при $n \gg 1$ приближается к $1 + \alpha/2$).

### Ответ

$\mathbb{E}[\text{успешный поиск}] = 1 + \frac{n-1}{2m} = \Theta(1 + \alpha/2)$.

## Задача 6. Открытая адресация: обнаружение ошибки

В реализации хеш-таблицы с открытой адресацией и линейным зондированием коллега написал удаление так: при удалении ключа $k$ просто обнуляем слот (ставим `EMPTY`). Покажите пример, когда это приводит к ошибке. Как исправить реализацию?

### Решение

Пусть $m = 5$, $h(k) = k \bmod 5$. Вставляем ключи в порядке: $5$ (слот 0), $10$ (слот 0 занят → слот 1), $3$ (слот 3).

Состояние таблицы: `[5, 10, _, _, 3]` (индексы 0–4).

Теперь удаляем ключ $5$, обнуляя слот 0: `[_, 10, _, _, 3]`.

Ищем ключ $10$: $h(10) = 0$, слот 0 — `EMPTY`. Линейное зондирование интерпретирует `EMPTY` как "цепочка зондирования закончилась, ключ не найден". Возвращается `false`, хотя ключ $10$ есть в слоте 1.

**Исправление — метод "захоронения" (tombstone).** Вместо `EMPTY` используется специальный маркер `DELETED`:

- При вставке: `DELETED`-слот трактуется как свободный (можно вставить).
- При поиске: `DELETED`-слот пропускается (зондирование продолжается).
- При поиске: только настоящий `EMPTY`-слот прерывает зондирование.

Таким образом, после удаления $5$ таблица: `[DELETED, 10, _, _, 3]`. Поиск $10$: слот 0 — `DELETED` (пропускаем), слот 1 — $10$ (нашли).

**Недостаток:** накопление `DELETED`-маркеров замедляет поиск. Периодически нужно перехешировать таблицу, убирая `DELETED`.

### Ответ

Обнуление разрывает цепочку зондирования. Решение: маркер `DELETED`, который пропускается при поиске, но используется при вставке.

## Задача 7. Строковый хеш и коллизии

Два студента предложили хеш-функции для строк:
- Студент A: $h_A(s) = \bigoplus_{i} s[i]$ (XOR всех символов)
- Студент B: $h_B(s) = \left(\sum_{i=0}^{n-1} s[i] \cdot 31^i\right) \bmod (10^9+7)$

Докажите, что $h_A$ неудовлетворительна: найдите два разных слова, дающих одинаковый $h_A$, и объясните структурную причину этого недостатка. Почему $h_B$ лучше?

### Решение

**Контрпример для $h_A$:** слова `"ab"` и `"ba"`:

$$
h_A(\text{"ab"}) = \text{'a'} \oplus \text{'b'} = 97 \oplus 98 = 3
$$
$$
h_A(\text{"ba"}) = \text{'b'} \oplus \text{'a'} = 98 \oplus 97 = 3
$$

Это не исключение, а правило: для любой перестановки символов строки XOR-хеш одинаков, так как XOR коммутативен и ассоциативен. Следовательно, все $k!$ перестановок строки длины $k$ с различными символами хешируются в одно значение — колоссальное количество коллизий.

Ещё один пример: `"aa"` и `""` — обе дают $h_A = 0$ (XOR числа с самим собой = 0, XOR пустой строки = 0). Аналогично, `"aabb"` = `"ab"` = `"ba"` = ... — любые строки с каждым символом чётное число раз.

**Структурная причина:** $h_A$ не учитывает *позицию* символа. Это нарушает требование равномерности: строки длины $n$ образуют классы эквивалентности по мультимножеству символов, и весь класс хешируется в одну точку.

**Почему $h_B$ лучше:** каждый символ умножается на уникальный вес $31^i$, зависящий от позиции. Перестановка символов даёт разные хеши (так как $31^i \neq 31^j$ при $i \neq j$). $h_B$ — это значение полинома $P(31) \bmod M$, и для случайных строк оно ведёт себя близко к равномерному распределению на $\{0, \ldots, M-1\}$.

### Ответ

$h_A$("ab") = $h_A$("ba") = 3. Причина: XOR не зависит от порядка. $h_B$ взвешивает позиции, избегая этой проблемы.

## Задача 8. Подсчёт анаграмм

Дан массив строк. Сгруппируйте анаграммы вместе (строки, состоящие из одинаковых символов в разном порядке). Реализуйте решение за $O(N \cdot L \log L)$, где $N$ — число строк, $L$ — средняя длина строки. Используйте `std::unordered_map`.

### Решение

Ключевая идея: две строки — анаграммы тогда и только тогда, когда их отсортированные версии совпадают. Используем отсортированную строку как ключ хеш-таблицы, значение — список исходных строк.

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <unordered_map>
#include <algorithm>

std::vector<std::vector<std::string>>
group_anagrams(const std::vector<std::string>& strs) {
    std::unordered_map<std::string, std::vector<std::string>> groups;

    for (const std::string& s : strs) {
        std::string key = s;
        std::sort(key.begin(), key.end());  // O(L log L)
        groups[key].push_back(s);           // O(L) для хеша ключа
    }

    std::vector<std::vector<std::string>> result;
    result.reserve(groups.size());
    for (auto& [key, group] : groups) {
        result.push_back(std::move(group));
    }
    return result;
}

int main() {
    std::vector<std::string> input = {"eat", "tea", "tan", "ate", "nat", "bat"};
    auto groups = group_anagrams(input);
    for (const auto& group : groups) {
        for (const auto& s : group) std::cout << s << " ";
        std::cout << "\n";
    }
    // Возможный вывод:
    // eat tea ate
    // tan nat
    // bat
    return 0;
}
```

**Анализ сложности:**
- Для каждой из $N$ строк: сортировка $O(L \log L)$, хеширование ключа $O(L)$, итого $O(L \log L)$ на строку.
- Общее: $O(N \cdot L \log L)$.
- Память: $O(N \cdot L)$ для хранения всех строк.

**Альтернативный ключ.** Вместо сортировки можно использовать массив частот символов (для ASCII/латиницы). Это $O(L)$ на строку, итого $O(NL)$ — быстрее, но ключ менее универсален.

### Ответ

Ключ = отсортированная строка. Сложность $O(NL \log L)$. Все анаграммы с одним ключом попадают в одну группу.

## Задача 9. Подмассив с нулевой суммой

Дан массив целых чисел $a[0..n-1]$. Найдите подмассив $a[l..r]$ с нулевой суммой (или сообщите, что такого нет). Реализуйте за $O(n)$.

### Решение

Используем префиксные суммы и хеш-таблицу. Обозначим $P[i] = a[0] + a[1] + \cdots + a[i-1]$ (с $P[0] = 0$). Сумма подмассива $a[l..r] = P[r+1] - P[l]$. Сумма равна нулю тогда и только тогда, когда $P[r+1] = P[l]$ для некоторых $l \leq r+1$.

Значит, задача сводится к: найти два одинаковых значения в массиве $P[0], P[1], \ldots, P[n]$.

```cpp
#include <iostream>
#include <vector>
#include <unordered_map>
#include <optional>

std::optional<std::pair<int,int>>
zero_sum_subarray(const std::vector<int>& a) {
    int n = static_cast<int>(a.size());
    // seen[prefix_sum] = индекс, где эта сумма впервые встретилась
    std::unordered_map<int, int> seen;
    seen[0] = 0;  // P[0] = 0 (пустой префикс)

    int prefix = 0;
    for (int i = 0; i < n; ++i) {
        prefix += a[i];
        // P[i+1] = prefix
        auto it = seen.find(prefix);
        if (it != seen.end()) {
            // P[i+1] = P[it->second], значит sum(a[it->second .. i]) = 0
            return std::make_pair(it->second, i);
        }
        seen[prefix] = i + 1;
    }
    return std::nullopt;
}

int main() {
    std::vector<int> a = {3, 4, -7, 3, 1, 3, 1, -4, -2, -2};
    auto result = zero_sum_subarray(a);
    if (result) {
        auto [l, r] = *result;
        std::cout << "Подмассив [" << l << ", " << r << "]\n";
        // a[0..2] = 3+4-7 = 0
    } else {
        std::cout << "Не найдено\n";
    }

    std::vector<int> b = {1, 2, 3};
    auto result2 = zero_sum_subarray(b);
    std::cout << (result2.has_value() ? "Найдено" : "Не найдено") << "\n";
    return 0;
}
```

**Трассировка для $a = [3, 4, -7]$:** $P = [0, 3, 7, 0]$. При $i=2$: prefix=0, seen[0]=0. Возвращаем $(0, 2)$: подмассив $a[0..2] = 3+4-7=0$.

**Анализ:** один проход по массиву, каждый шаг — O(1) работа с хеш-таблицей. Итого O(n).

### Ответ

Хранить в хеш-таблице первое вхождение каждой префиксной суммы. Если сумма повторилась — найден нулевой подмассив. Сложность $O(n)$.

## Задача 10. Проектирование: LRU Cache

Реализуйте структуру данных LRU Cache (Least Recently Used) с операциями `get(key)` и `put(key, value)`, обе за $O(1)$ в среднем. Размер кэша ограничен $C$ элементами; при добавлении нового элемента в полный кэш вытесняется наименее недавно использованный.

### Решение

Классическое решение: комбинация `std::unordered_map` (для O(1) доступа по ключу) и двусвязного списка (для O(1) перемещения элемента в начало и удаления последнего).

- Список хранит элементы в порядке от "самого недавнего" (голова) до "самого старого" (хвост).
- Хеш-таблица: ключ → итератор на узел списка.
- `get(key)`: найти в таблице, переместить узел в начало списка. O(1).
- `put(key, value)`: если ключ есть — обновить, переместить в начало. Если нет — вставить в начало, при переполнении удалить хвост и его запись из таблицы. O(1).

```cpp
#include <iostream>
#include <list>
#include <unordered_map>
#include <stdexcept>

class LRUCache {
public:
    explicit LRUCache(int capacity) : cap_(capacity) {}

    int get(int key) {
        auto it = map_.find(key);
        if (it == map_.end()) return -1;  // не найдено
        // Перемещаем в начало (самый недавний)
        cache_.splice(cache_.begin(), cache_, it->second);
        return it->second->second;
    }

    void put(int key, int value) {
        auto it = map_.find(key);
        if (it != map_.end()) {
            it->second->second = value;
            cache_.splice(cache_.begin(), cache_, it->second);
            return;
        }
        // Новый ключ
        if (static_cast<int>(cache_.size()) == cap_) {
            // Вытесняем LRU (хвост)
            auto lru = cache_.back();
            map_.erase(lru.first);
            cache_.pop_back();
        }
        cache_.emplace_front(key, value);
        map_[key] = cache_.begin();
    }

private:
    int cap_;
    // Список пар (key, value), от новейшего к старейшему
    std::list<std::pair<int,int>> cache_;
    // Ключ -> итератор в cache_
    std::unordered_map<int, std::list<std::pair<int,int>>::iterator> map_;
};

int main() {
    LRUCache lru(2);
    lru.put(1, 10);
    lru.put(2, 20);
    std::cout << lru.get(1) << "\n";   // 10 (1 становится недавним)
    lru.put(3, 30);                    // вытесняет 2 (LRU)
    std::cout << lru.get(2) << "\n";   // -1 (вытеснен)
    std::cout << lru.get(3) << "\n";   // 30
    lru.put(4, 40);                    // вытесняет 1 (LRU)
    std::cout << lru.get(1) << "\n";   // -1 (вытеснен)
    std::cout << lru.get(3) << "\n";   // 30
    std::cout << lru.get(4) << "\n";   // 40
    return 0;
}
```

**Почему это O(1)?**
- `std::unordered_map::find` — O(1) в среднем.
- `std::list::splice` — O(1) (перелинковка указателей, без копирования).
- `std::list::pop_back` — O(1).

Без хеш-таблицы нельзя найти элемент за O(1); без двусвязного списка нельзя за O(1) вставить в начало и удалить хвост. Именно их комбинация даёт нужную сложность.

### Ответ

Комбинация `unordered_map<key, list::iterator>` и `list<pair<key,value>>`. `get` и `put` — O(1) в среднем благодаря хеш-таблице для доступа и списку для порядка.
