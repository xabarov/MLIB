## Задача 1. Количество листьев в дереве решений

Для сортировки $n = 4$ элементов нарисуйте дерево решений и найдите минимальное количество листьев. Чему равна минимально возможная высота такого дерева (то есть минимальное гарантированное число сравнений в худшем случае)?

### Решение

Для $n = 4$ элементов существует $4! = 24$ различных перестановки. Каждый лист дерева решений соответствует одной перестановке, поэтому дерево должно содержать **не менее 24 листьев**.

Бинарное дерево высоты $h$ имеет не более $2^h$ листьев:

$$
2^h \ge 24 \implies h \ge \lceil \log_2 24 \rceil = \lceil 4{,}585 \rceil = 5
$$

Значит, в худшем случае потребуется **не менее 5 сравнений**.

Для проверки: алгоритм сортировки вставками на 4 элементах делает ровно 5 сравнений в худшем случае ($1 + 2 + 2 = 5$ при оптимальной реализации бинарного поиска для вставки). Нижняя оценка достигается.

### Ответ

Минимум 24 листа, минимальная высота 5. В худшем случае нужно **не менее 5 сравнений**.

## Задача 2. Трассировка пузырьковой сортировки

Проследите шаг за шагом работу пузырьковой сортировки с флагом на массиве $[4, 2, 7, 1, 3]$. Укажите после каждого прохода состояние массива и количество обменов.

### Решение

**Проход 1** (сравниваем соседние j=0..3):

| j | Сравнение | Обмен? | Массив |
|---|---|---|---|
| 0 | 4 > 2 | Да | [2, 4, 7, 1, 3] |
| 1 | 4 > 7 | Нет | [2, 4, 7, 1, 3] |
| 2 | 7 > 1 | Да | [2, 4, 1, 7, 3] |
| 3 | 7 > 3 | Да | [2, 4, 1, 3, 7] |

После прохода 1: `[2, 4, 1, 3, 7]`, обменов = 3, swapped = true.

**Проход 2** (j=0..2):

| j | Сравнение | Обмен? | Массив |
|---|---|---|---|
| 0 | 2 > 4 | Нет | [2, 4, 1, 3, 7] |
| 1 | 4 > 1 | Да | [2, 1, 4, 3, 7] |
| 2 | 4 > 3 | Да | [2, 1, 3, 4, 7] |

После прохода 2: `[2, 1, 3, 4, 7]`, обменов = 2, swapped = true.

**Проход 3** (j=0..1):

| j | Сравнение | Обмен? | Массив |
|---|---|---|---|
| 0 | 2 > 1 | Да | [1, 2, 3, 4, 7] |
| 1 | 2 > 3 | Нет | [1, 2, 3, 4, 7] |

После прохода 3: `[1, 2, 3, 4, 7]`, обменов = 1, swapped = true.

**Проход 4** (j=0..0):

| j | Сравнение | Обмен? | Массив |
|---|---|---|---|
| 0 | 1 > 2 | Нет | [1, 2, 3, 4, 7] |

swapped = false, **останавливаемся**.

Итого: 4 прохода, 6 обменов.

### Ответ

Результат: `[1, 2, 3, 4, 7]`. Выполнено 4 прохода и 6 обменов. Флаг ускорил завершение на последнем проходе.

## Задача 3. Число инверсий и сортировка вставками

Докажите, что количество обменов соседних элементов при сортировке вставками равно числу инверсий в исходном массиве. Подсчитайте число инверсий в массиве $[3, 1, 4, 1, 5, 9, 2, 6]$.

### Решение

**Доказательство.** При сортировке вставками на шаге $i$ элемент `key = a[i]` сдвигается влево до тех пор, пока не встретит элемент, не превосходящий его. Каждый сдвиг на одну позицию влево соответствует устранению одной инверсии: пара $(j, i)$ с $a[j] > \text{key}$ и $j < i$ — это ровно одна инверсия. Поскольку каждый сдвиг устраняет ровно одну инверсию и не создаёт новых, а алгоритм завершается, когда инверсий не остаётся, суммарное число сдвигов равно числу инверсий.

**Подсчёт инверсий в $[3, 1, 4, 1, 5, 9, 2, 6]$:**

Перебираем все пары $(i, j)$ с $i < j$ и $a[i] > a[j]$:

- 3 > 1 (позиции 0,1) — инверсия
- 3 > 1 (позиции 0,3) — инверсия
- 3 > 2 (позиции 0,6) — инверсия
- 4 > 1 (позиции 2,3) — инверсия
- 4 > 2 (позиции 2,6) — инверсия
- 5 > 2 (позиции 4,6) — инверсия
- 9 > 2 (позиции 5,6) — инверсия
- 9 > 6 (позиции 5,7) — инверсия

Итого: **8 инверсий**.

### Ответ

Число обменов в сортировке вставками = числу инверсий. В массиве $[3, 1, 4, 1, 5, 9, 2, 6]$ ровно **8 инверсий**.

## Задача 4. Рекуррентность merge sort

Решите рекуррентность $T(n) = 2T(n/2) + cn$ с начальным условием $T(1) = d$. Выведите явную формулу. Какое значение имеет глубина рекурсии?

### Решение

Развернём рекуррентность методом итерации. На $k$-м уровне рекурсии:
- Имеется $2^k$ подзадач размера $n/2^k$.
- Каждая подзадача вносит вклад $c \cdot (n/2^k)$ в шаг слияния.
- Суммарная работа на уровне $k$: $2^k \cdot c \cdot (n/2^k) = cn$.

Рекурсия продолжается до тех пор, пока размер подзадачи не достигнет 1: $n/2^k = 1 \implies k = \log_2 n$.

Глубина рекурсии: $\log_2 n$ уровней, на каждом — работа $cn$. На уровне $\log_2 n$ имеется $n$ подзадач размера 1, каждая стоит $d$:

$$
T(n) = cn \cdot \log_2 n + dn = \Theta(n \log n)
$$

Явная формула: $T(n) = cn \log_2 n + dn$.

### Ответ

$T(n) = cn \log_2 n + dn = \Theta(n \log n)$. Глубина рекурсии равна $\lfloor \log_2 n \rfloor$.

## Задача 5. Худший случай быстрой сортировки

Для алгоритма quickSort (Ломуто, pivot = последний элемент) опишите класс входных данных, приводящих к $O(n^2)$. Как рандомизация устраняет эту проблему?

### Решение

**Плохие входные данные:**

1. **Отсортированный массив** $[1, 2, 3, \ldots, n]$. Pivot = $n$ (максимум), все остальные элементы уходят в левую часть. Размеры подзадач: $n-1$ и $0$. Рекуррентность: $T(n) = T(n-1) + O(n)$, итого $O(n^2)$.

2. **Обратно отсортированный массив** $[n, n-1, \ldots, 1]$. Pivot = $1$ (минимум), все уходят вправо. Та же рекуррентность.

3. **Массив из одинаковых элементов** $[k, k, \ldots, k]$. Pivot = $k$, все элементы $\le k$, левая часть размером $n-1$. Снова $O(n^2)$.

**Рандомизация:** перед разбиением выбираем pivot как `a[rand(lo, hi)]` и меняем его с `a[hi]`. Теперь для любого фиксированного входа распределение опорных элементов случайно. Ожидаемое число сравнений:

$$
E[T(n)] = 2n \ln n + O(n) = O(n \log n)
$$

Вероятность того, что конкретный запуск займёт $\Omega(n^2)$ при $n$ элементах, равна $O(1/n!)$ — пренебрежимо мала.

### Ответ

Худший случай — отсортированный (или обратно отсортированный) массив при выборе крайнего элемента как pivot. Рандомизация переводит задачу в вероятностный анализ и гарантирует $O(n \log n)$ в ожидании для любого входа.

## Задача 6. Подсчёт инверсий за O(n log n)

Дан массив $[5, 3, 1, 4, 2]$. Подсчитайте число инверсий, модифицировав алгоритм сортировки слиянием. Покажите, как пары считаются при каждом слиянии.

### Решение

Запускаем модифицированный mergeSort и считаем, сколько раз элемент из правой половины «перепрыгивает» через элементы левой при слиянии.

**Разбиение:** $[5, 3, 1, 4, 2] \to [5, 3] \mid [1, 4, 2]$

**Левая половина $[5, 3]$:**
- Слияние $[5] \mid [3]$: 3 < 5, прыжок через 1 элемент → **+1 инверсия**. Результат: [3, 5].

**Правая половина $[1, 4, 2]$:**
- Слияние $[1] \mid [4]$: 1 < 4, прыжка нет → **+0**. Результат: [1, 4].
- Слияние $[1, 4] \mid [2]$: 
  - 1 < 2, берём 1 → +0.
  - 2 < 4, прыжок через 1 элемент → **+1 инверсия**. Берём 2.
  - Берём 4 → +0.
  - Результат: [1, 2, 4]. Итого: **+1 инверсия**.

**Итоговое слияние $[3, 5] \mid [1, 2, 4]$:**
- 1 < 3, прыжок через 2 элемента (3 и 5) → **+2 инверсии**. Берём 1.
- 2 < 3, прыжок через 2 элемента (3 и 5) → **+2 инверсии**. Берём 2.
- 3 < 4, берём 3 → +0.
- 4 < 5, прыжок через 1 элемент → **+1 инверсия**. Берём 4.
- Берём 5 → +0.
- Итого: **+5 инверсий**.

**Итог:** $1 + 0 + 1 + 5 = 7$ инверсий.

**Проверка прямым перебором:** пары $(5,3), (5,1), (5,4), (5,2), (3,1), (3,2), (4,2)$ — ровно 7.

### Ответ

Число инверсий в $[5, 3, 1, 4, 2]$ равно **7**.

## Задача 7. Сортировка связного списка

Какой из алгоритмов — быстрая сортировка или сортировка слиянием — лучше подходит для сортировки однонаправленного связного списка? Обоснуйте, используя свойства доступа к памяти.

### Решение

**Быстрая сортировка — плохой выбор для связного списка:**
- Разбиение Ломуто требует произвольного доступа к элементам по индексу (swap(a[i], a[j])). В связном списке доступ к $k$-му узлу занимает $O(k)$, что делает разбиение неэффективным.
- Нет возможности быстро найти «средний» элемент для выбора pivot.

**Сортировка слиянием — естественный выбор:**
- Разделение списка пополам реализуется через алгоритм «черепаха и заяц» (slow/fast pointer) за $O(n)$ без дополнительной памяти.
- Слияние двух отсортированных списков — классическая операция, работающая за $O(n)$ и не требующая произвольного доступа.
- Не нужна дополнительная память на вспомогательный массив (в отличие от слияния массивов) — можно перешивать указатели на месте.

**Итоговая сложность:** $O(n \log n)$ по времени, $O(\log n)$ по стеку вызовов (глубина рекурсии), $O(1)$ по дополнительной памяти.

Именно поэтому `std::list::sort` в C++ использует сортировку слиянием.

### Ответ

Для связного списка **сортировка слиянием** предпочтительнее: она не требует произвольного доступа, естественно разделяет список через slow/fast pointer и сливает без дополнительной памяти.

## Задача 8. Нижняя оценка для частичного порядка

Докажите, что если известно, что входной массив является перестановкой $\{1, \ldots, n\}$ и гарантированно содержит не более $k$ инверсий, то задача сортировки этого массива решается за $O(n + k \log k)$ сравнений. Для каких $k$ это лучше $O(n \log n)$?

### Решение

**Алгоритм:**

1. Запускаем сортировку вставками. По задаче 3 она делает ровно $k$ сравнений/сдвигов при $k$ инверсиях.

Однако $O(n + k)$ достигается только если вставки адаптивны — это неочевидно. Рассмотрим другой подход:

2. Разделим массив на блоки длиной $\approx k$. Поскольку инверсий всего $k$, каждый элемент смещён от своей целевой позиции не более чем на $k$. Найдём все «неправильные» элементы за $O(n)$, их не более $2k$. Отсортируем их за $O(k \log k)$ и вставим на место за $O(n)$.

**Итого:** $O(n + k \log k)$.

**Сравнение с $O(n \log n)$:**

$$
O(n + k \log k) \le O(n \log n) \iff k \log k = O(n \log n)
$$

При $k = O(n)$ это даёт $O(n \log n)$, выигрыша нет. Но при $k = O(n / \log n)$:

$$
k \log k = O\!\left(\frac{n}{\log n} \cdot \log \frac{n}{\log n}\right) = O\!\left(\frac{n}{\log n} \cdot \log n\right) = O(n)
$$

Итого $O(n)$ — существенно лучше $O(n \log n)$.

### Ответ

При $k = O(n / \log n)$ инверсиях сортировка выполняется за $O(n)$ — асимптотически лучше нижней оценки $\Omega(n \log n)$ для общего случая.

## Задача 9. Разбиение Ломуто: трассировка

Выполните разбиение Ломуто на массиве $[3, 6, 8, 10, 1, 2, 1]$ с pivot = последний элемент ($= 1$). Укажите состояние массива и значения $i$, $j$ на каждом шаге. Определите финальную позицию pivot.

### Решение

Начальное состояние: `a = [3, 6, 8, 10, 1, 2, 1]`, `pivot = a[6] = 1`, `i = -1`.

| j | a[j] | a[j] ≤ 1? | i после | Массив после шага |
|---|---|---|---|---|
| 0 | 3 | Нет | -1 | [3, 6, 8, 10, 1, 2, 1] |
| 1 | 6 | Нет | -1 | [3, 6, 8, 10, 1, 2, 1] |
| 2 | 8 | Нет | -1 | [3, 6, 8, 10, 1, 2, 1] |
| 3 | 10 | Нет | -1 | [3, 6, 8, 10, 1, 2, 1] |
| 4 | 1 | Да | 0 | swap(a[0], a[4]): **[1, 6, 8, 10, 3, 2, 1]** |
| 5 | 2 | Нет | 0 | [1, 6, 8, 10, 3, 2, 1] |

Конец цикла. Финальный swap: `swap(a[i+1], a[6])` = `swap(a[1], a[6])`:

$$
[1, \mathbf{1}, 8, 10, 3, 2, \mathbf{6}]
$$

Pivot находится на позиции **1**.

- Левый подмассив (позиции 0..0): `[1]` — все элементы $\le 1$.
- Правый подмассив (позиции 2..6): `[8, 10, 3, 2, 6]` — все элементы $> 1$.

### Ответ

После разбиения: `[1, 1, 8, 10, 3, 2, 6]`. Pivot (значение 1) на позиции **1**.

## Задача 10. Синтез: оптимальная сортировка на практике

Вам нужно отсортировать массив из $10^7$ целых чисел в диапазоне $[0, 10^6]$. Предложите оптимальную стратегию. Сравните по времени и памяти с `std::sort`.

### Решение

**Анализ задачи:**
- $n = 10^7$ элементов.
- Диапазон значений $k = 10^6 + 1$.
- Отношение $n/k = 10$ — каждое значение встречается в среднем 10 раз.

**Стратегия 1: Сортировка подсчётом (Counting Sort)**

```
1. Создать массив count[0..10^6] = 0.         // O(k) памяти, O(k) времени
2. Пройти по массиву a: count[a[i]]++.        // O(n) времени
3. Восстановить массив a по count.            // O(n + k) времени
```

- Время: $O(n + k) = O(10^7 + 10^6) = O(10^7)$.
- Память: $O(k) = O(10^6)$ на массив подсчёта — примерно 4 МБ для `int`.
- Устойчивость: да (при аккуратной реализации с prefix sum).

**Стратегия 2: std::sort (introsort)**

- Время: $O(n \log n) = O(10^7 \cdot 23) \approx 2.3 \cdot 10^8$ операций.
- Память: $O(\log n)$ стека.
- На практике с константой $\approx 3$ нс/операцию: $\approx 0.7$ сек.

**Сравнение:**

| | Counting Sort | std::sort |
|---|---|---|
| Время | $O(n + k) \approx 10^7$ | $O(n \log n) \approx 2 \cdot 10^8$ |
| Память | $O(k) = 4$ МБ | $O(\log n) = O(1)$ |
| Применимость | Только целые в диапазоне $[0, k]$ | Любые сравнимые объекты |

**Вывод:** при данных параметрах сортировка подсчётом в $\approx 20$ раз быстрее по числу операций. Расход 4 МБ дополнительной памяти обычно приемлем. Если диапазон был бы $[0, 10^9]$, понадобилось бы 4 ГБ — тогда лучше использовать radix sort (8 проходов по байту) или `std::sort`.

### Ответ

Оптимальна **сортировка подсчётом**: время $O(n + k) \approx O(10^7)$, память $O(k) \approx 4$ МБ. Она примерно в 20 раз быстрее `std::sort` по числу операций. При большом диапазоне — radix sort или `std::sort`.
