# SHAD Interactive: план этапа "карта курса и trace-механика"

Этот этап должен превратить `SHAD/interactive` из набора хороших отдельных
миссий в понятный учебный маршрут. Сейчас приложение уже доказывает, что
математику полезно трогать руками: ядро, определитель, матрица и перестановки
дают разные игровые ощущения. Следующий вопрос другой: понимает ли пользователь,
зачем проходить их в таком порядке и как этот язык переносится на алгоритмы,
программирование и анализ данных.

## Общая оценка текущего состояния

Направление выглядит сильным. Главное, что уже получилось:

- **Интерактив не дублирует лекции.** Лучшие сцены объясняют через действие:
  подвинуть вектор, схлопнуть площадь, собрать перестановку транспозициями.
- **Есть собственный тон.** Бумажная мастерская, чертежные линии, теплые
  акценты и Меби отличают приложение от обычного образовательного dashboard.
- **Архитектура уже выдерживает рост.** `MissionShell`, model modules,
  registry/navigation, progress store, Vitest и Playwright smoke дают нормальную
  основу для новых механик.
- **Меби начинает работать как проводник.** Персонаж держит эмоциональную
  обратную связь и может стать маркером инварианта, а не просто украшением.
- **Mobile теперь пригоден для прохождения.** Drawer и числовые inputs в
  матричной миссии сильно снижают хрупкость управления.

При этом продукт пока ощущается скорее как мастерская с несколькими станками,
а не как учебное приключение. Пользователь попадает в текущую миссию, но еще не
видит общую карту: что уже открыто, что дальше, какие идеи собраны и как они
соотносятся с лекциями.

## Чего не хватает сильнее всего

### 1. First screen как карта, а не случайная миссия

Сейчас root route уводит сразу в `Охоту за ядром`. Это удобно для разработки,
но слабее как учебный продукт: новичок не понимает, что в приложении уже есть
несколько маршрутов и что между ними есть смысловая цепочка.

Нужен route `#/` или `#/map`, где видно:

- доступные миссии;
- рекомендуемый порядок;
- статус прохождения;
- связанную лекцию;
- следующий лучший шаг.

### 2. Игровой прогресс между миссиями

Инвариант-ключи есть внутри shell, но между миссиями они пока не становятся
историей пользователя. Нужна маленькая мета-структура:

- сколько тем открыто;
- какие идеи собраны;
- что откроется после следующей миссии;
- куда ведет next mission.

Это не должно превращаться в тяжелую gamification-систему. Достаточно понятной
карты и одного приятного момента награды.

### 3. Первый настоящий trace вне геометрии

`TraceStep` уже добавлен как контракт, но пока он не ожил в интерфейсе. Чтобы
приложение не застряло в линейной алгебре, следующая миссия должна показать
состояние алгоритма по шагам.

Лучший кандидат на этот этап: **Графовый диспетчер: BFS/DFS**.

Почему именно он:

- в `SHAD/combinatorics/4. Graphs/lesson.md` уже есть богатый материал по
  графам, связности, деревьям, эйлеровым путям и ориентации;
- графы естественно играются: вершины, ребра, очередь/стек, посещенные узлы;
- BFS/DFS хорошо показывает trace: текущее состояние, стоимость, инвариант;
- эта механика потом переносится на топосортировку, SCC, Дейкстру и DP.

### 4. Более выразительная награда

После успеха уже появляются выводы, но момент "я сделал" пока спокойный.
Нужна короткая, не шумная reward-анимация:

- вспышка/поворот инвариант-ключа;
- unlock следующего уровня/миссии;
- короткий итог собранной идеи;
- реакция Меби, связанная с конкретным инвариантом.

### 5. Дизайн-система карты и trace

Сейчас сцены выглядят согласованно, но карте курса и trace-миссиям нужны свои
устойчивые паттерны:

- `MissionCard` для карты;
- `CourseMap` / `CoursePath` для связей между темами;
- `TracePanel` для очереди, стека, visited set, стоимости;
- `StateToken` для маленьких состояний вроде `queued`, `visited`, `current`;
- `UnlockToast` или встроенный reward-блок без модальных окон.

## Продуктовая цель этапа

Сделать так, чтобы пользователь за 30 секунд понял:

1. какие игровые миссии уже есть;
2. какую пройти первой;
3. чему каждая миссия учит;
4. что откроется дальше;
5. что этот формат подходит не только для геометрии, но и для алгоритмов.

## Приоритеты

### P0. Course Map route

Добавить страницу карты курса.

Рекомендуемые файлы:

- `src/pages/CourseMapPage.tsx`;

- `src/game/components/CoursePath.tsx`;

Минимальная карта:

```text
Подстановки -> Матрицы -> Определитель -> Ядро
                         -> будущие графы
```

Важно: порядок можно спорить методически, но в интерфейсе нужна осмысленная
цепочка. Для первого варианта достаточно показать "рекомендуемый следующий
шаг" из `MissionDefinition.nextMissionRoute`.

Готово, когда при открытии `#/` пользователь видит все playable/prototype
миссии, прогресс и кнопку перехода к следующей миссии.

### P0. Progress summary между миссиями

Расширить визуальный прогресс без усложнения store:

- показывать собранные инвариант-ключи по миссиям;
- выводить `completed / total` для каждой mission card;
- показывать 1 строку "собранная идея" после полного прохождения;
- подсвечивать next recommended mission.

Вероятно, текущего `progressStore` достаточно. Если нужно, добавить selectors,
а не новую сущность состояния.

Готово, когда карта курса обновляется после прохождения уровней без ручного
refresh и без дублирования логики в компонентах миссий.

### P0. Trace UI primitives

Сделать компоненты, которые будут нужны BFS/DFS и будущему программированию:

- `TracePanel`;
- `TraceStepList`;
- `QueueStackView`;
- `StateToken`;
- `InvariantCheck`.

Они должны принимать данные, близкие к текущему контракту:

```ts
type TraceStep = {
  id: string
  label: string
  state: Record<string, unknown>
  cost?: number
  invariantOk?: boolean
}
```

Готово, когда можно показать trace без знания конкретного алгоритма в общем
компоненте.

### P1. Mission "Графовый диспетчер"

Добавить первую алгоритмическую mission.

Рабочая тема: BFS/DFS на небольшом графе.

Игровой цикл:

1. Есть граф из 6-8 вершин.
2. Пользователь выбирает следующую вершину из очереди или стека.
3. Приложение показывает `current`, `visited`, `frontier`, `edges checked`.
4. Уровень проверяет инвариант:
   - BFS: очередь хранит границу по слоям;
   - DFS: стек/рекурсия идет в глубину;
   - связность: все достижимые вершины посещены;
   - дерево обхода не содержит лишних ребер.

Минимальные уровни:

- `bfs-layers`: пройти BFS и сохранить порядок слоев;
- `dfs-stack`: пройти DFS стеком;
- `connected-component`: найти компоненту связности;
- `repair-trace`: исправить неверный следующий шаг.

Рекомендуемые файлы:

- `src/visualizations/graphs/graphTraceModel.ts`;
- `src/visualizations/graphs/graphTraceModel.test.ts`;
- `src/visualizations/graphs/GraphDispatcherMission.tsx`;
- запись в `missions.ts`, `missionRegistry.ts`, `routeLoaders.ts`,
  `navigation.ts`;
- ссылка из `SHAD/combinatorics/4. Graphs/lesson.md`.

Готово, когда миссия ощущается как пошаговый алгоритм, а не как статический
рисунок графа.

### P1. Reward polish

Добавить аккуратное усиление момента успеха:

- короткая CSS-анимация ключа в `RewardMeter`;
- unlock-highlight у следующего уровня;

Готово, когда прохождение уровня визуально считывается без чтения текста, но
не превращается в аттракцион.

## UX-направление

Карта курса должна быть не landing page и не маркетинговый hero. Первый экран -
рабочая учебная карта:

- слева/сверху краткий прогресс;
- в центре цепочка миссий;
- карточки компактные, без вложенных cards;
- у каждой карточки есть статус, собранные ключи, связанная лекция и кнопка
  запуска;
- planned missions видны как приглушенные станции маршрута.

Trace-миссия должна ощущаться как диспетчерская:

- граф занимает главный экран;
- очередь/стек и visited set видны рядом как рабочие инструменты;
- стоимость и инвариант показываются badge'ами;
- ошибка объясняется через состояние, а не длинным текстом;
- на mobile trace controls идут после графа и остаются крупными для tap.

## Риски

- Если начать с красивой карты без новой trace-миссии, этап улучшит упаковку,
  но не расширит тип учебного опыта.
- Если начать с большой алгоритмической платформы, можно утонуть в абстракциях.
  Поэтому trace primitives должны родиться из BFS/DFS, а не из общего видения
  всех будущих алгоритмов.
- Если карта станет landing page, мы потеряем рабочую плотность приложения.
  Это должна быть карта действий, а не презентация возможностей.
