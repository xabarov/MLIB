# Правила написания `lesson.md` и `qa.md` в `SHAD/`

Этот файл — практический guide по созданию новых тем в `./SHAD/` на основе уже сложившегося формата репозитория и реального опыта написания темы про векторные пространства.

Здесь собраны:

- шаблон структуры темы;
- правила для `lesson.md`;
- правила для `qa.md`;
- правила по LaTeX и Markdown;
- правила по картинкам и анимациям;
- рекомендуемый workflow проверки перед коммитом.

---

## 1. Как устроена тема в `SHAD/`

Обычно одна тема живёт в отдельной папке. Примеры:

- `SHAD/algebra/6_Matrices/`
- `SHAD/algebra/7_Vector_spaces/`
- `SHAD/mathematical_analysis/5_derivative/`

Минимальный набор файлов для новой темы:

- `lesson.md` — лекция;
- `qa.md` — 10 задач с подробными решениями.

Опционально:

- `practice.md` — листок без решений;
- `qa_shad.md` — версия задач ближе к вступительному стилю;
- `assets/` — картинки, GIF, SVG, локальные визуалы;
- `generate_visuals.py` — воспроизводимая генерация картинок и анимаций.

---

## 2. Базовый принцип

Материал должен быть одновременно:

- понятным человеку, который быстро повторяет тему;
- достаточно строгим для подготовки к ШАД;
- единообразным относительно соседних тем;
- хорошо рендериться в Markdown-просмотрщике без “съехавшего” LaTeX.

Идеальный стиль — это не конспект в одну строку и не академический учебник на 50 страниц, а объясняющая лекция с хорошей структурой и задачник с подробными, но не водянистыми решениями.

---

## 3. Правила для `lesson.md`

### 3.1. Обязательная верхняя структура

Рекомендуемый каркас:

```md
# Лекция: <тема>

## План

1. ...
2. ...
3. ...

---

## 1. <первый раздел>
...
```

В лекции почти всегда должны быть:

- определения;
- интуиция;
- простые примеры;
- ключевые теоремы/факты;
- хотя бы несколько “рабочих” вычислительных примеров;
- типичные ошибки;
- блок “что важно для поступления в ШАД”;
- итог;
- вопросы для самопроверки.

### 3.2. Что должно быть в хорошем разделе

Каждый большой раздел лучше строить так:

1. короткая постановка идеи;
2. определение или формулировка;
3. объяснение смысла;
4. пример;
5. при необходимости — доказательство или алгоритм.

### 3.3. Оптимальная глубина

Нужно писать так, чтобы:

- сильный читатель мог быстро пробежать глазами;
- средний читатель мог реально понять тему с нуля;
- материал не распухал от повторов.

Если формально можно сказать в одну строчку, но без пояснения это будет непонятно, лучше добавить короткую интерпретацию.

Если идея уже была объяснена дважды, третий раз повторять не надо.

## 4. Правила для `qa.md`

### 4.1. Базовый формат

Рекомендуемый заголовок:

```md
# 10 задач в стиле вступительного экзамена ШАД  
по теме: <тема>
```

Дальше:

- короткий вводный абзац;
- 10 задач;
- у каждой задачи:
  - условие;
  - `### Решение`;
  - `### Ответ`;
- в конце можно добавить `<details>` с краткой сводкой, что именно тренируют эти задачи.

### 4.2. Баланс задач

Хороший `qa.md` должен покрывать разные уровни:

1. базовая проверка определения;
2. простое вычисление;
3. задача на базовый структурный факт;
4. задача с параметром;
5. задача на размерность / ранг / число степеней свободы;
6. задача на координаты / преобразование / базис;
7. задача на доказательство;
8. задача на типичную ловушку;
9. задача в “полуэкзаменационном” стиле;
10. задача на синтез нескольких идей.

### 4.3. Что важно в решениях

Решение должно:

- идти по шагам;
- не пропускать ключевые переходы;
- не быть переусложнённым;
- заканчиваться коротким ответом.

Плохо:

```md
### Решение
Очевидно.
```

Тоже плохо:

```md
### Решение
<страница текста без структуры>
```

Хорошо:

- разбить на шаги;
- назвать случаи;
- отделить вычисление от вывода;
- в конце дать короткий итог.

## 5. Стиль изложения

### 5.1. Тон

Тон должен быть:

- спокойный;
- объясняющий;
- математически аккуратный;
- без разговорной небрежности.

### 5.2. Хорошие формулировки

Предпочтительно:

- “Следовательно,”
- “Значит,”
- “Покажем, что…”
- “Рассмотрим…”
- “Это важно, потому что…”
- “Идея состоит в том, что…”

### 5.3. Чего избегать

- “ну”, “просто”, “как бы”;
- чрезмерной патетики;
- фраз вроде “это супер важно” без пояснения, почему именно;
- длинных абзацев без структуры.

## 6. Правила по LaTeX и Markdown

Это критически важная часть. Именно здесь чаще всего ломается рендер.

### 6.1. Что использовать

- inline-формулы: `$...$`
- блочные формулы: `$$ ... $$`
- системы: `\begin{cases} ... \end{cases}`
- матрицы: `\begin{pmatrix} ... \end{pmatrix}`
- выравнивание длинных формул: `\begin{aligned} ... \end{aligned}`

### 6.2. Главное правило: не делать `$$` внутри списков с отступом

Плохо:

```md
1. Формула
   $$
   \dim(U+W)=\dim U+\dim W-\dim(U\cap W)
   $$
```

Во многих рендерах это может “съехать”.

Лучше так:

```md
1. Формула $\dim(U+W)=\dim U+\dim W-\dim(U\cap W)$.
```

Или так:

```md
Формула:
$$
\dim(U+W)=\dim U+\dim W-\dim(U\cap W).
$$
```

### 6.3. В списках предпочтительнее inline-формулы

Если формула короткая, внутри списка лучше писать её inline:

```md
- размерность равна $\dim U$;
- пересечение тривиально: $U\cap W=\{0\}$.
```

### 6.4. Для длинных матричных или многострочных выражений лучше `aligned`

Если формула многоступенчатая, лучше:

```md
$$
\begin{aligned}
v&=S[v]_f\\
&=
\begin{pmatrix}
1 & 1\\
1 & -1
\end{pmatrix}
\cdot
\begin{pmatrix}
a\\
b
\end{pmatrix}\\
&=
\begin{pmatrix}
a+b\\
a-b
\end{pmatrix}.
\end{aligned}
$$
```

Это обычно стабильнее, чем “рваные” куски на отдельных строках.

### 6.5. Явное умножение матриц лучше писать через `\cdot`

Не обязательно, но для Markdown-рендеров полезно:

```md
\begin{pmatrix}
1 & 1\\
1 & -1
\end{pmatrix}
\cdot
\begin{pmatrix}
a\\
b
\end{pmatrix}
```

Так формула визуально понятнее и реже выглядит “слипшейся”.

### 6.6. Не усложнять там, где можно проще

Если выражение короткое, лучше не тащить отдельный блочный LaTeX:

Вместо:

```md
Пусть
$$
U\cap W=\{0\}.
$$
```

лучше:

```md
Пусть $U\cap W=\{0\}$.
```

### 6.7. Пустые строки вокруг блочных формул

Желательно оставлять пустую строку:

- до блочной формулы;
- после блочной формулы.

Это снижает риск странного рендера.

### 6.8. Что обязательно перепроверять глазами

После написания нужно отдельно просмотреть:

- все `pmatrix`;
- все `cases`;
- все `aligned`;
- формулы внутри списков;
- формулы рядом с картинками и подписями.

## 7. Правила по картинкам и анимациям

### 7.1. Когда стоит добавлять визуалы

Иллюстрации особенно полезны, если тема содержит:

- геометрический смысл;
- переход между координатами / базисами;
- движение / деформацию;
- сумму, пересечение, проекцию, разложение;
- размерность, ядро, образ, подпространства;
- зависимость от параметра;
- алгоритм, который удобно показывать пошагово.

Если картинка не объясняет идею лучше текста, она не нужна.

### 7.2. Как выбрать инструмент

Выбор инструмента зависит от типа визуала:

| Нужно | Основной путь | Когда применять |
|---|---|---|
| Точная схема, график, геометрия, анимация параметра | `generate_visuals.py` рядом с темой | если важны координаты, подписи, воспроизводимость |
| Обложка/hero/метафора без точных чисел | `generate_images.py` + `SHAD/lecture_visual_generation/` | если нужна editorial-картинка в едином стиле |
| Статичный “плакатный” PNG/PDF | skill `canvas-design` | если нужна композиция уровня постера/слайда, а не математический график |
| Генеративная графика / p5.js / параметрический sketch | skill `algorithmic-art` | если визуал сам является алгоритмом или интерактивной заготовкой |
| React/Three.js интерактив | `SHAD/interactive` + skills `frontend-design`, `webapp-testing` | если студент должен вращать сцену, менять режимы, видеть динамику |

Правило: **точные математические рисунки сначала делать кодом**. LLM/image skills использовать для визуального языка, hero-иллюстраций, метафор и сложной композиции, но не для фактических диаграмм с числами.

### 7.3. Что лучше использовать на практике

На текущий момент для этой кодовой базы хорошо подходят:

- `matplotlib` — основа для статических рисунков;
- `seaborn` — стили и аккуратные палитры;
- `plotly` — если нужен интерактив или экспорт через `kaleido`;
- `pillow` и `imageio` — сборка GIF-анимаций;
- `manim` — опционально, если в окружении доступны системные зависимости.

Практический минимальный стек:

```bash
source .venv/bin/activate
python -m pip install matplotlib seaborn plotly kaleido pillow imageio
```

### 7.4. Куда складывать визуалы

Для темы:

- `assets/*.png`
- `assets/*.gif`

Рядом с темой можно хранить:

- `generate_visuals.py`

То есть структура такая:

```text
SHAD/algebra/<topic>/
  lesson.md
  qa.md
  assets/
    figure_1.png
    concept.gif
  generate_visuals.py
```

### 7.5. Как вставлять в `lesson.md`

Использовать относительные пути:

```md
![Подпись](assets/figure.png)
```

или

```md
![Анимация перехода между базисами](assets/basis_change.gif)
```

Для интерактивной модели использовать ссылку на статическую сборку:

```md
[Интерактивная модель](../../interactive/dist/index.html#/algebra/linear-maps/kernel)
```

Если `dist/` не коммитится, такая ссылка должна поддерживаться CI/deploy-процессом или явно пересобираться перед публикацией.

### 7.6. Правила для самих картинок

Картинка должна:

- объяснять ровно одну идею;
- иметь читаемые цвета;
- не быть перегруженной;
- содержать подпись в тексте рядом;
- быть достаточно контрастной на светлом фоне.

### 7.7. Цвета

Хорошая практика:

- один и тот же объект держать одним цветом по всей теме;
- использовать 2-4 основных цвета;
- не полагаться только на цвет, если можно добавить подпись.

Например:

- базис 1 — синий;
- базис 2 — фиолетовый;
- итоговый вектор — красный;
- вспомогательные построения — оранжевый.

Для SHAD-визуалов предпочтительно синхронизироваться с палитрой из
`SHAD/lecture_visual_generation/lecture_visual_design_system.md`:

- фон `#faf9f5`;
- чернила/контур `#141413`;
- вторичный серый `#b0aea5`;
- мягкие панели `#e8e6dc`;
- акценты: `#d97757`, `#6a9bcc`, `#788c5d`, `#7c6ccf`.

### 7.8. Когда лучше GIF, а не PNG

GIF полезен, когда нужно показать:

- изменение базиса;
- движение точки;
- разложение вектора на компоненты;
- изменение параметра;
- пошаговое построение объекта.

PNG полезен, когда нужен:

- стабильный рисунок;
- схема;
- плоскость / линия / пересечение;
- итоговая геометрическая картинка без движения.

### 7.9. Важное правило: визуалы должны быть воспроизводимыми

Нельзя просто положить “случайную” картинку без способа её пересоздать.

Если добавляются визуалы, желательно хранить рядом генератор:

- `generate_visuals.py`

Он должен:

- читать только локальные зависимости;
- писать файлы в `assets/`;
- быть воспроизводимым;
- не требовать ручной магии после запуска.

Минимальный шаблон для `generate_visuals.py`:

```python
from pathlib import Path

import matplotlib.pyplot as plt

ROOT = Path(__file__).resolve().parent
ASSETS = ROOT / "assets"

COLORS = {
    "bg": "#faf9f5",
    "ink": "#141413",
    "gray": "#b0aea5",
    "orange": "#d97757",
    "blue": "#6a9bcc",
    "green": "#788c5d",
    "purple": "#7c6ccf",
}


def ensure_assets() -> None:
    ASSETS.mkdir(exist_ok=True)


def save_figure(fig: plt.Figure, filename: str) -> None:
    fig.savefig(ASSETS / filename, dpi=180, bbox_inches="tight", facecolor=COLORS["bg"])
    plt.close(fig)


def main() -> None:
    ensure_assets()
    fig, ax = plt.subplots(figsize=(8, 4.5), facecolor=COLORS["bg"])
    ax.set_facecolor(COLORS["bg"])
    ax.axhline(0, color=COLORS["gray"], linewidth=1)
    ax.axvline(0, color=COLORS["gray"], linewidth=1)
    ax.plot([0, 1], [0, 1], color=COLORS["orange"], linewidth=3)
    ax.set_aspect("equal", adjustable="box")
    ax.axis("off")
    save_figure(fig, "example.png")


if __name__ == "__main__":
    main()
```

### 7.10. LLM-изображения через `generate_images.py`

Для hero/метафор использовать `generate_images.py`; стиль по умолчанию берётся из
`SHAD/lecture_visual_generation/lecture_visual_prompt_suffix.txt`.

Пример dry-run:

```bash
.venv/bin/python generate_images.py \
  --dry-run \
  --jobs SHAD/lecture_visual_generation/lecture_images.example.json
```

Пример пакетной генерации в assets темы:

```bash
.venv/bin/python generate_images.py \
  --jobs SHAD/algebra/13_Complex_spaces/lecture_image_jobs.json \
  --out-dir SHAD/algebra/13_Complex_spaces/assets
```

В prompt обязательно указывать, что на изображении не должно быть длинного текста,
фейковых осей, выдуманных метрик и водяных знаков. Если нужен точный график,
использовать `generate_visuals.py`, а не LLM-картинку.

### 7.11. Как применять skills

Рекомендуемый набор Codex skills для этого репозитория:

- `doc-coauthoring` — структура лекции, guide, план работ, ревизия документации;
- `canvas-design` — статичные PNG/PDF-артефакты, когда нужна сильная композиция;
- `algorithmic-art` — p5.js/generative sketches, seeded randomness, flow fields;
- `frontend-design` — дизайн `SHAD/interactive` и UI-состояний;
- `webapp-testing` — Playwright-проверка dev/preview-сборок;
- `theme-factory` — единый визуальный стиль между документами/HTML/графикой;
- `pdf` — сборка или проверка PDF-версий материалов.

Skills не заменяют проверку содержания. После применения любого визуального skill
нужно убедиться, что результат:

- математически не врёт;
- соответствует теме;
- имеет понятный source/rebuild path;
- не ломает общий визуальный язык SHAD.

### 7.12. Проверка `SHAD/interactive`

Интерактивные сцены лежат в `SHAD/interactive`. Базовая проверка:

```bash
cd SHAD/interactive
npm ci
npm run lint
npm run build
npm run dev -- --host 127.0.0.1
```

Открыть маршрут:

```text
http://127.0.0.1:5173/#/algebra/linear-maps/kernel
```

Для Playwright-проверок использовать Python из `.venv` и системный Chromium:

```python
from playwright.sync_api import sync_playwright

with sync_playwright() as p:
    browser = p.chromium.launch(
        headless=True,
        executable_path="/snap/bin/chromium",
        args=["--no-sandbox", "--disable-dev-shm-usage"],
    )
    page = browser.new_page(viewport={"width": 1440, "height": 960})
    page.goto("http://127.0.0.1:5173/#/algebra/linear-maps/kernel")
    page.wait_for_selector("canvas")
    assert page.locator("canvas").count() >= 1
    browser.close()
```

Перед коммитом интерактива проверить минимум desktop и mobile viewport:

- нет горизонтального overflow;
- canvas не пустой и не сжат до узкой колонки;
- нет console errors;
- текст не перекрывает кнопки и сцену;
- ссылка из `lesson.md` ведёт на существующий hash-route.

## 10. Чеклист перед завершением темы

Перед тем как считать тему готовой, пройтись по этому списку.

### По содержанию

- Есть чёткий план лекции.
- Все ключевые определения сформулированы.
- Есть интуиция, а не только формальности.
- Есть примеры.

- Файлы лежат в правильной папке.

- Если есть визуалы, то есть `assets/`.

## 11. Рекомендуемый workflow

Практически удобно делать так:

1. Посмотреть 2-3 соседние темы того же раздела.
2. Создать папку темы.
3. Сначала написать полный `lesson.md` без визуалов.
4. Затем написать `qa.md`.
5. Потом пройтись по LaTeX и убрать потенциально опасные конструкции.
6. Если тема геометрическая или визуальная, добавить `assets/` и `generate_visuals.py`.
7. Встроить картинки рядом с соответствующими разделами лекции.
8. Ещё раз визуально открыть Markdown и проверить рендер.

## 12. Короткая памятка

Если нужно запомнить только главное, то вот оно:

- одна тема = одна папка;
- минимум: `lesson.md` и `qa.md`;
- лекция должна объяснять, а не просто перечислять факты;
- задачник должен тренировать, а не только повторять лекцию;
- формулы в списках лучше писать inline;
- сложные матричные формулы лучше оформлять через `aligned`;
- картинки и GIF полезны, если реально объясняют идею;
- визуалы должны быть воспроизводимыми через скрипт.
